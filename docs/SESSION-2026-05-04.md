- Created `/root/.config/opencode/plugin/claude-bridge.ts` (~320 lines). Reads `~/.claude/settings.json`. Maps opencode tool names to Claude tool names (`bash→Bash`, `read→Read`, `edit→Edit`, `multiedit→Edit`, `patch→Edit`, `glob→Glob`, `grep→Grep`, `webfetch→WebFetch`, `websearch→WebSearch`, `task→Agent`, `todowrite→TaskUpdate`, `todoread→TaskList`). Module exports `{ id: "claude-bridge", server: plugin }` (the `id` field is required by opencode for path-loaded plugins; missing it caused the initial plugin-load failure that was fixed in this session).
- Created `/root/.config/opencode/opencode.json` wiring the plugin via `"plugin": ["./plugin/claude-bridge.ts"]`.
- Created `/root/.config/opencode/plugin/package.json` with type-only dep on `@opencode-ai/plugin`.
- Validated: `opencode debug config` shows registered plugin, scope global. `opencode serve` log shows successful plugin load after the `id` fix. Created `/tmp/bridge-test.py` smoke test that confirmed 8/8 deny-rule matcher cases pass against real `settings.json`, and 4/4 hook-spawn cases (Read .env→deny, Read main.ts→allow, Bash env→deny, Bash ls→allow) pass against real `policy-block.sh`.
